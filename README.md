# Manta

Manta is a small, systems-style programming language implemented in Rust. It features manual memory management, `defer`-based cleanup, enum pattern matching, and a Go-inspired syntax. The compiler produces native binaries by emitting LLVM IR via [Inkwell](https://github.com/TheDan64/inkwell) and linking with `cc`.

```manta
mod main

// Div is the return enum for the div function
// and encodes both success and failure states
type Div enum {
  Ok(i64)
  DivByZero
}

fn div(a, b i64) Div {
  if b == 0 {
    // this returns the Div.DivByZero enum value but
    // the type can be inferred
    return .DivByZero
  }
  
  return .Ok(a / b)
}

fn main() i64 {
  mut a = 5
  // mutable bindings can be updated
  a += 5
  
  // immutable bindings can not
  let result = div(2, 4)
  match result {
    .Ok(_) { print("success!") }
    .DivByZero { print("tried to divide by zero") }
  }
  
  // identifiers can be shadowed, result here is rebound to a new value
  // via a let pattern which work like match patterns
  let .Ok(result) = div(10, 20) or {
    // if the result was not the .Ok variant, the block will run instead
    print("division failed")
    return
  }
  
  // result is available here though we're not using it for anything
  result
  
  // this will panic if the result is not .Ok
  let .Ok(_) = div(100, 0) !
}
```

## Project goals

- Minimal, explicit systems-language with manual memory management (no GC).
- Familiar surface syntax for Rust/C/Zig/Go users: `fn`, `let`, `defer`, `new`/`free`, `match`, `try`/`catch`.
- Small, practical grammar designed to be easy to learn

## Building

This project is a Rust crate. To build the compiler locally you need a Rust toolchain (rustc + cargo). On macOS using Homebrew or rustup is recommended.

```bash
cargo build           # debug build
cargo build --release # release build
```

## Testing

Tests are snapshot tests auto-generated by `build.rs`. For each `.manta` source file in `tests/src/`, test functions are generated that compare compiler output (JSON) against expected files in `tests/{lexer,parser,noder,blocker,codegen}/`. Codegen outputs are `.ll` files and are the actual LLVM IR geneated by the pipeline. Each stage validates its own output independently.

```bash
cargo test --all             # run all tests
cargo test <test_name>       # run a single test, e.g. cargo test test_noder_let_stmt
```

To add a new test: add a `.manta` file to `tests/src/` and the corresponding expected JSON files in the relevant stage directories.

Example `.manta` programs used as test inputs live in `tests/src/`. They demonstrate features like `defer`/`free`, enum pattern matching, `let`-or unwrapping, pointer types, control flow (`loop`/`while`/`for`-in), `break`/`continue`, ranges (`..<`, `..=`), and `mut` bindings.

## Project structure

The compiler pipeline is:

```
Source → Lexer → Parser → AST → Noder → HIR → Typer → Blocker → MIR → LLVM IR → Object File → Binary
```

**Stages:**
- **Lexer** (`src/parser/lexer.rs`): Tokenizes source into `Token`/`TokenKind`.
- **Parser** (`src/parser.rs`, `src/parser/`): Recursive descent with parselets for expressions. Produces an `AST` (enums `Decl`, `Expr`, `Stmt`, `Pattern`).
- **Noder** (`src/noder.rs`): Lowers AST → HIR (`NodeTree`). Each construct becomes a `Node` enum variant identified by `NodeID`.
- **Typer** (`src/noder/typer.rs`): Type checks and infers types over the `NodeTree`, populating a `type_map` side table (`NodeID → TypeSpec`).
- **Blocker** (`src/blocker.rs`): Lowers HIR → MIR, building a control-flow graph of `BasicBlock`s with SSA-like `ValueId`s.
- **Codegen** (`src/codegen.rs`, `src/codegen/`): Lowers MIR → LLVM IR using Inkwell. Runs target-specific optimizations, emits an object file, and links to a native binary via clang.

**Key data structures:**
- `NodeTree`: Holds all HIR nodes, `type_map`, and `symbol_map` side tables.
- `SideTable<K,V>`: BTreeMap + Vec store used for `type_map` (`NodeID → TypeSpec`) and `symbol_map` (`SymID → NodeID`).
- `TypeSpec`: Type information including `Any` (unknown), `IntLiteral(i64)`/`FloatLiteral(f64)` (unresolved numeric literals), `InferredEnumExpr/Pat` (partial enum info), `Panic` (unreachable), `Unit` (void).
- `StrStore` / `StrID`: String interning — all identifiers are stored once and referenced by integer ID.
- `MirFunction`: Contains `LocalId`-indexed locals, `BlockId`-indexed `BasicBlock`s, and an entry block.

## Language spec references

Primary design documents are in `docs/`. Key files:

- `docs/language_spec.md` — high-level language features, examples, and semantics.
- `docs/grammar.ebnf` — starting grammar for the parser (EBNF draft).
- `docs/token_list.md` — the token kinds the lexer emits.

These documents were used to author the test programs in `tests/src/` and should be considered the source-of-truth while the language evolves.

## Proposals

`proposals/` contains informal explorations of potential language directions — traits, explicit lifetimes, zero-value semantics, Go-style switch expressions, and more. These are **not** part of the language and should not be used as a reference for current behavior. They exist as a scratchpad for iterating on syntax and features before committing to implementation.

## Command-line interface

> **Note:** `manta check`, `manta run`, and `manta fmt` are currently stubs — they parse flags but do not yet perform real work. `manta build` is implemented for files but can not yet build modules.

The CLI executable is named `manta` and supports the following subcommands:

### `manta build`

Compiles a `.manta` source file to a native binary. Requires `clang` to be installed and on `PATH` for linking.

```bash
./target/release/manta build hello.manta
./target/release/manta build hello.manta -o hello
```

Flags:
- `-o, --out-file <OUT_FILE>` — output binary name (defaults to the input file's stem)

### `manta check`

Runs checks and reports results without producing build artifacts. *(stub)*

```bash
./target/release/manta check
```

### `manta run`

Builds and immediately executes the program. *(stub)*

```bash
./target/release/manta run
```

### `manta fmt`

Formats source files using the standard Manta formatter. *(stub)*

```bash
./target/release/manta fmt src/
./target/release/manta fmt -w src/   # write changes in-place
```

Flags:
- `-w, --write` — write changes to files in-place (otherwise prints diffs)
- positional `FILES...` — files or directories to format

### Global flags

- `-v` / `--verbose` — increase verbosity (stackable; `-vv` for more verbose output)
- `-p, --path <PATH>` — path to the project or file to operate on (defaults to current directory)
